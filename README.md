## 1. Введение

В современном образовательном процессе одной из ключевых задач является контроль посещаемости учебных занятий. Традиционные методы учёта, такие как бумажные журналы или электронные таблицы, обладают существенными недостатками: они трудоёмки, подвержены ошибкам, не позволяют оперативно получать статистику и анализировать данные. Кроме того, при большом количестве групп и студентов обработка таких данных становится крайне затруднительной, а обеспечение их целостности и сохранности — практически невозможной без специализированного программного обеспечения.

В связи с этим возникла потребность в создании специализированной информационной системы, которая позволила бы автоматизировать процесс учёта посещаемости, сделать его более прозрачным и удобным для преподавателей. Такая система должна не только фиксировать факт присутствия студента на занятии, но и предоставлять надёжное хранение данных, обеспечивать разграничение доступа и формировать основу для дальнейшего анализа посещаемости.

Целью данной производственной практики являлась разработка серверной части такой системы — backend API, который обеспечивает хранение, обработку и предоставление данных о группах, студентах, занятиях и посещаемости. В задачи входило проектирование схемы базы данных, реализация REST API с полным набором методов для работы клиентского приложения, обеспечение аутентификации преподавателей, документирование API и подготовка инфраструктуры для развёртывания.

## 2. Анализ предметной области и постановка задачи

Прежде чем приступить к разработке, был проведён анализ существующего прототипа серверной части системы и схемы базы данных. На основании изучения исходного кода и документации были выявлены ключевые проблемы, определившие направление дальнейшей работы.

Исходный прототип API (AttendanceApi) представлял собой приложение с реализованными, но нестабильными методами. Схема базы данных включала семнадцать таблиц со сложной иерархией связей: факультеты, кафедры, направления подготовки, профили, базовые и рабочие группы, учебные годы, зачисления, дисциплины, типы занятий, учебные планы, аудитории, сессии посещаемости и статусы. Такая структура, хотя и была спроектирована с учётом полной модели образовательного процесса, оказалась избыточной для задач минимально жизнеспособного продукта. Многие таблицы, например справочники факультетов, кафедр, направлений и аудиторий, не использовались в основных сценариях работы преподавателя и лишь усложняли как серверную логику, так и клиентскую интеграцию.

Ещё одной существенной проблемой являлось отсутствие единого формата ответов API. Различные методы возвращали данные в разных структурах, что затрудняло работу фронтенд-разработчиков и приводило к необходимости реализовывать специальную логику обработки для каждого эндпоинта. Документация API полностью отсутствовала, что делало невозможной эффективную параллельную работу над клиентской и серверной частями.

Были также выявлены проблемы с обработкой ошибок: при некорректных запросах сервер зачастую возвращал необработанные исключения вместо информативных сообщений об ошибке. Миграции базы данных и тестовые данные находились в неконсистентном состоянии, что затрудняло развёртывание системы с нуля.

Проанализировав все выявленные проблемы, была сформулирована основная задача: превратить набор разрозненных компонентов в стабильную, документированную и готовую к интеграции серверную часть. Для этого необходимо было упростить схему базы данных, сохранив всю функциональность, критичную для MVP, привести API к единому формату, реализовать все недостающие методы, обеспечить надёжную обработку ошибок и подготовить исчерпывающую документацию.

Ключевым требованием со стороны клиентского приложения было обеспечение следующих сценариев: авторизация преподавателя в системе, просмотр списка групп и студентов, создание занятий с автоматическим формированием записей посещаемости, редактирование статусов посещаемости отдельных студентов, а также возможность управления группами и студентами. Все эти сценарии должны были поддерживаться стабильными и предсказуемыми методами API.

## 3. Выбор технологического стека

Для реализации серверной части приложения был выбран технологический стек, обеспечивающий высокую производительность, удобство разработки и совместимость с клиентской частью, построенной на JavaScript-экосистеме. Основой стала платформа Node.js, которая позволяет использовать единый язык программирования на клиенте и сервере, облегчая взаимодействие между членами команды и повторное использование кода.

Использование TypeScript в качестве основного языка программирования было продиктовано необходимостью обеспечить надёжность и предсказуемость серверного кода. В backend-разработке статическая типизация особенно важна, поскольку ошибки типов в серверном коде могут приводить к потере данных или нарушению безопасности. TypeScript помогает выявлять такие ошибки на этапе компиляции, делает код более самодокументированным и упрощает рефакторинг, что критично при работе со сложными моделями данных.

В качестве веб-фреймворка был выбран Express пятой версии — широко используемый и проверенный временем фреймворк для Node.js. Express предоставляет минималистичный, но гибкий набор инструментов для создания HTTP-серверов: маршрутизацию, middleware для обработки запросов, поддержку различных форматов данных. Его архитектура позволяет легко расширять функциональность за счёт подключения дополнительных модулей, при этом не навязывая жёсткую структуру проекта.

Для работы с базой данных была выбрана ORM-библиотека Sequelize в связке с расширением sequelize-typescript. Этот выбор обусловлен несколькими факторами. Во-первых, Sequelize является одной из наиболее зрелых и стабильных ORM для Node.js с обширной документацией и активным сообществом. Во-вторых, расширение sequelize-typescript позволяет описывать модели данных с помощью декораторов, что делает код более декларативным и приближает его к описанию схемы базы данных. В-третьих, Sequelize предоставляет встроенную систему миграций, позволяющую версионировать изменения схемы базы данных и воспроизводить их в любом окружении.

В качестве системы управления базами данных был выбран PostgreSQL пятнадцатой версии. PostgreSQL является одной из наиболее надёжных и функциональных реляционных СУБД с открытым исходным кодом. Он обеспечивает поддержку сложных типов данных, в том числе перечислений (ENUM), которые активно используются в проекте для хранения статусов посещаемости. Кроме того, PostgreSQL обладает мощными средствами обеспечения целостности данных — внешними ключами с каскадными операциями, ограничениями уникальности и транзакционной безопасностью.

Для аутентификации преподавателей была реализована схема на основе JSON Web Token. Библиотека jsonwebtoken обеспечивает генерацию и верификацию токенов, а bcryptjs — безопасное хеширование паролей с использованием алгоритма bcrypt. Такой подход позволяет реализовать stateless-аутентификацию, при которой серверу не нужно хранить информацию о текущих сессиях, что упрощает масштабирование.

Документирование API было выполнено с помощью связки swagger-jsdoc и swagger-ui-express. Первая библиотека позволяет описывать спецификацию OpenAPI непосредственно в коде проекта, а вторая — предоставлять интерактивный веб-интерфейс для просмотра и тестирования методов API. Наличие такой документации значительно ускоряет разработку клиентской части, поскольку фронтенд-разработчики получают исчерпывающую информацию о доступных методах, форматах запросов и ответов.

Для контейнеризации и упрощения развёртывания была использована связка Docker и Docker Compose. Многоступенчатая сборка Docker-образа позволяет минимизировать итоговый размер контейнера, а Docker Compose обеспечивает оркестрацию нескольких сервисов — базы данных, миграций и самого приложения — с помощью одной команды.

## 4. Проектирование серверной части

Прежде чем приступить к написанию кода, была тщательно продумана архитектура серверного приложения. Ключевым решением на этапе проектирования стало радикальное упрощение схемы базы данных. Исходные семнадцать таблиц были сокращены до пяти, при этом вся функциональность, необходимая для MVP, была полностью сохранена.

Обоснованием для такого решения послужил анализ реальных сценариев работы преподавателя. Для повседневного учёта посещаемости преподавателю не требуется информация об организационной структуре университета — факультетах, кафедрах и направлениях подготовки. Ему необходимо знать только название группы, список студентов в ней и иметь возможность создавать занятия и отмечать посещаемость. Таким образом, вся иерархия справочных таблиц была заменена единой таблицей групп с простым текстовым названием, например «ПИ-221».

Аналогичным образом была упрощена модель занятий. Вместо сложной цепочки из таблиц дисциплин, типов занятий, учебных планов, аудиторий и сессий посещаемости была создана единая таблица занятий, содержащая все необходимые атрибуты: группу, название предмета, дату и время проведения, подгруппу и количество академических часов. Название предмета хранится непосредственно в записи занятия, что исключает необходимость в отдельных справочниках дисциплин.

Статусы посещаемости, ранее представлявшие отдельную таблицу, были заменены перечислением (ENUM) с четырьмя фиксированными значениями: «присутствует», «отсутствует», «опоздал» и «уважительная причина». Этот набор статусов покрывает все реальные ситуации учёта посещаемости и не требует динамического расширения, что делает использование перечисления более эффективным решением по сравнению с отдельной таблицей.

Итоговая схема базы данных состоит из пяти таблиц. Таблица `lecturer` хранит данные преподавателей: имя, фамилию, отчество, логин и хешированный пароль. Таблица `student_group` содержит учебные группы с уникальными названиями. Таблица `student` хранит данные студентов с привязкой к группе и номером подгруппы от одного до трёх. Таблица `lesson` описывает занятия: привязка к группе и преподавателю, название предмета, дата и время проведения, необязательный номер подгруппы и количество академических часов. Наконец, таблица `attendance` фиксирует посещаемость — связь между занятием и студентом со статусом из перечисления.

Все связи между таблицами реализованы через внешние ключи с каскадным удалением. Это означает, что при удалении группы автоматически удаляются все связанные студенты, занятия и записи посещаемости, что обеспечивает целостность данных без необходимости ручной очистки.

Структура серверного приложения была организована по функциональным слоям. Слой моделей описывает структуру данных и связи между таблицами с помощью декораторов Sequelize. Слой контроллеров содержит бизнес-логику обработки запросов. Слой маршрутов определяет соответствие HTTP-методов и URL-адресов конкретным контроллерам. Слой middleware обеспечивает сквозную функциональность — аутентификацию и обработку ошибок. Такое разделение упрощает навигацию по коду, тестирование отдельных компонентов и внесение изменений.

Формат ответов API был унифицирован. Все успешные ответы содержат поле `success` со значением `true` и поле `data` с запрашиваемыми данными. Все ответы об ошибках содержат поле `success` со значением `false` и поле `message` с описанием ошибки на русском языке. Это позволяет клиентскому приложению использовать единую логику обработки ответов вне зависимости от конкретного метода.

## 5. Реализация серверной логики

В процессе реализации особое внимание уделялось тому, чтобы API был не только функциональным, но и предсказуемым, устойчивым к ошибкам и удобным для интеграции с клиентским приложением. Было реализовано семнадцать эндпоинтов, покрывающих все сценарии работы преподавателя.

Первым был реализован модуль аутентификации. При входе в систему преподаватель отправляет логин и пароль, сервер проверяет их корректность, сравнивая введённый пароль с хешем в базе данных с помощью алгоритма bcrypt, и в случае успеха генерирует JWT-токен с идентификатором преподавателя. Токен имеет ограниченный срок действия — двадцать четыре часа, после чего требуется повторная аутентификация. Middleware аутентификации автоматически проверяет наличие и валидность токена для всех защищённых маршрутов, извлекая из него данные текущего пользователя.

Модуль управления группами предоставляет полный набор CRUD-операций. При создании группы преподаватель может передать не только название, но и массив студентов, которые будут автоматически добавлены в группу. Это существенно ускоряет первоначальную настройку системы. При создании проверяется уникальность названия группы, и в случае дублирования возвращается информативное сообщение об ошибке с кодом состояния 409. Получение списка групп включает вложенную информацию о студентах, что позволяет клиентскому приложению отображать полную картину одним запросом.

Модуль управления студентами позволяет добавлять, редактировать и удалять студентов, а также получать список студентов конкретной группы с возможностью фильтрации по подгруппе. Фильтрация реализована через параметр запроса, что позволяет клиентскому приложению динамически формировать списки для отображения на странице занятия. Валидация данных предусматривает проверку обязательных полей — имени, фамилии и принадлежности к группе, а также корректности номера подгруппы, который должен быть в диапазоне от одного до трёх.

Центральным элементом системы является модуль занятий. При создании занятия преподаватель указывает группу, название предмета, дату и время проведения, а опционально — подгруппу и количество академических часов. После создания записи занятия система автоматически формирует записи посещаемости для всех студентов соответствующей группы или подгруппы. Каждый студент получает начальный статус «присутствует». Такой подход реализует сценарий, при котором преподаватель по умолчанию считает всех студентов присутствующими и затем отмечает только исключения — отсутствующих, опоздавших или тех, кто предоставил уважительную причину. Это минимизирует количество действий, которые преподаватель должен совершить для типичного занятия, где присутствует большинство студентов.

Получение информации об отдельном занятии включает вложенные данные посещаемости с информацией о каждом студенте. Это позволяет клиентскому приложению отобразить полную картину занятия — список студентов с их текущими статусами — за один запрос к серверу, что положительно сказывается на скорости работы интерфейса.

Список занятий для текущего преподавателя отсортирован по дате и времени в обратном порядке, что обеспечивает отображение ближайших занятий в начале списка. Фильтрация по преподавателю выполняется автоматически на основании данных из JWT-токена, и каждый преподаватель видит только свои занятия.

Модуль посещаемости предоставляет метод изменения статуса конкретной записи. При обновлении проверяется, что переданный статус является одним из допустимых значений перечисления. Также реализован вспомогательный метод, возвращающий список доступных статусов с их русскоязычными наименованиями, что позволяет клиентскому приложению динамически формировать элементы интерфейса на основе данных от сервера.

Централизованная обработка ошибок реализована через middleware, перехватывающий все необработанные исключения. Для ожидаемых ошибок используется кастомный класс AppError, позволяющий указать HTTP-код ответа и человекочитаемое описание проблемы. Непредвиденные ошибки обрабатываются с кодом 500 и общим сообщением, при этом полная информация об ошибке записывается в серверный лог для последующей диагностики. Такой подход гарантирует, что клиент всегда получает структурированный ответ, а не сырое сообщение об исключении.

## 6. Интеграция с базой данных

Работа с базой данных является фундаментом серверного приложения, поскольку именно на этом уровне обеспечивается сохранность и целостность всех данных системы. Интеграция с PostgreSQL была выполнена через ORM Sequelize, что позволило абстрагировать SQL-запросы и работать с данными на уровне объектов.

Модели данных описаны с помощью декораторов библиотеки sequelize-typescript. Каждая модель соответствует одной таблице базы данных и описывает её поля, типы данных, ограничения и связи с другими моделями. Например, модель студента декларирует связь «принадлежит» с моделью группы через внешний ключ, а модель занятия декларирует связь «имеет много» с моделью посещаемости. Такое декларативное описание позволяет Sequelize автоматически формировать корректные SQL-запросы с необходимыми соединениями таблиц.

Управление схемой базы данных осуществляется через систему миграций. Была создана миграция, которая полностью перестраивает структуру базы данных: удаляет все семнадцать устаревших таблиц и создаёт пять новых с корректными типами данных, внешними ключами и ограничениями. Миграция поддерживает как применение, так и откат изменений, что позволяет безопасно вернуться к предыдущему состоянию базы данных в случае необходимости.

Для первоначального заполнения базы данных тестовыми данными был создан сидер. Он добавляет двух преподавателей с хешированными паролями, две учебные группы, восемь студентов с распределением по подгруппам, три занятия с различными параметрами и десять записей посещаемости с разными статусами. После вставки данных сидер корректирует последовательности автоинкремента для каждой таблицы, что предотвращает конфликты при последующем создании новых записей. Наличие тестовых данных позволяет сразу после развёртывания системы проверить её работоспособность и продемонстрировать функциональность.

Особое внимание было уделено каскадным операциям. При удалении группы автоматически удаляются все связанные студенты, а вместе с ними — все записи посещаемости этих студентов. Аналогично, при удалении занятия удаляются все связанные записи посещаемости. Это реализовано на уровне базы данных через механизм внешних ключей с директивой ON DELETE CASCADE, что гарантирует целостность данных вне зависимости от того, каким образом была инициирована операция удаления.

Конфигурация подключения к базе данных вынесена в переменные окружения. Это позволяет использовать одну и ту же кодовую базу в различных окружениях — локальной разработке, тестировании и продакшене — просто изменяя значения переменных. Для локальной разработки параметры подключения задаются в файле окружения, а при развёртывании в Docker-контейнерах они передаются через секцию environment в конфигурации Docker Compose.

Подготовка инфраструктуры развёртывания включала создание Docker-файла с многоступенчатой сборкой. На первом этапе выполняется компиляция TypeScript в JavaScript, на втором — формируется минимальный образ, содержащий только скомпилированный код и продакшен-зависимости. Docker Compose описывает три сервиса: базу данных PostgreSQL с проверкой готовности, одноразовый контейнер для выполнения миграций и сидов, а также контейнер приложения с автоматическим перезапуском. Благодаря этому весь стек разворачивается одной командой.

Документирование API реализовано в виде спецификации OpenAPI третьей версии. Спецификация описывает пятнадцать схем данных, покрывающих все объекты запросов и ответов, а также все семнадцать эндпоинтов с указанием параметров, тел запросов, кодов ответов и требований к аутентификации. Интерактивный веб-интерфейс Swagger UI доступен по адресу `/api-docs` и позволяет не только просматривать документацию, но и отправлять тестовые запросы непосредственно из браузера, что существенно ускоряет процесс интеграции с клиентской частью.

## 7. Результаты работы

В результате прохождения производственной практики была разработана полнофункциональная серверная часть системы учёта посещаемости учебных занятий. Сервер предоставляет REST API из семнадцати эндпоинтов, покрывающих все необходимые сценарии работы преподавателя.

Модуль аутентификации включает два метода: вход в систему с получением JWT-токена и получение профиля текущего преподавателя. Модуль управления группами предоставляет пять методов: получение списка всех групп, получение группы по идентификатору, создание группы с опциональным списком студентов, обновление названия группы и удаление группы. Модуль управления студентами включает четыре метода: получение студентов группы с фильтрацией по подгруппе, добавление студента, обновление данных студента и удаление студента. Модуль занятий предоставляет четыре метода: получение списка занятий текущего преподавателя, получение занятия с данными посещаемости, создание занятия с автоматической генерацией записей посещаемости и удаление занятия. Модуль посещаемости включает два метода: изменение статуса посещаемости и получение справочника статусов.

Все эндпоинты были протестированы вручную. Проверена корректная работа авторизации, защита маршрутов от неавторизованного доступа, валидация входных данных, обработка ошибок при обращении к несуществующим ресурсам, каскадное удаление связанных записей и корректность формата всех ответов. Компиляция TypeScript проходит без ошибок, что подтверждает типовую корректность всего кода.

Схема базы данных была упрощена с семнадцати таблиц до пяти без потери функциональности, критичной для MVP. Все связи между таблицами обеспечены внешними ключами с каскадными операциями, что гарантирует целостность данных. Миграции и сиды позволяют развернуть базу данных с тестовыми данными с нуля одной командой.

Подготовлена полная спецификация API в формате OpenAPI с интерактивным веб-интерфейсом Swagger UI. Настроена контейнеризация с помощью Docker и Docker Compose, позволяющая развернуть весь стек — базу данных, миграции и приложение — одной командой. Система успешно интегрирована с клиентским приложением, обеспечивая все необходимые сценарии работы: авторизацию, просмотр расписания, отметку и редактирование посещаемости, управление группами и студентами.

## 8. Заключение

Подводя итог производственной практике, можно с уверенностью сказать, что поставленные цели были достигнуты. Разработанная серверная часть системы учёта посещаемости обеспечивает надёжное хранение данных, предоставляет стабильный и документированный API для клиентских приложений и готова к дальнейшему развитию.

По сравнению с исходным прототипом, результат работы имеет ряд принципиальных улучшений. Схема базы данных упрощена более чем втрое — с семнадцати таблиц до пяти — при полном сохранении функциональности MVP. Все ответы API приведены к единому предсказуемому формату. Реализована полноценная обработка ошибок с информативными сообщениями. Подготовлена исчерпывающая документация Swagger. Настроено развёртывание через Docker.

В ходе работы были решены разнообразные задачи: анализ и рефакторинг существующей кодовой базы, проектирование новой схемы базы данных, реализация серверной логики с учётом требований клиентского приложения, обеспечение безопасности через JWT-аутентификацию и хеширование паролей, документирование и контейнеризация. Применение современных технологий — TypeScript, Express, Sequelize, PostgreSQL — позволило создать производительное и легко поддерживаемое серверное приложение.

Разработанная система может быть в дальнейшем расширена и дополнена. Например, можно реализовать агрегированную статистику посещаемости по группам и студентам, добавить экспорт данных в различные форматы, внедрить ролевую модель доступа для разных категорий пользователей, реализовать интеграцию с внешними системами управления учебным процессом. Однако уже в текущем виде серверная часть представляет собой полностью работоспособную и стабильную основу, обеспечивающую все критичные сценарии учёта посещаемости.

Прохождение производственной практики позволило не только применить на практике знания, полученные в процессе обучения, но и приобрести ценный опыт разработки реального серверного приложения. Были углублены навыки проектирования баз данных, разработки REST API, работы с ORM и системами контейнеризации, а также получено понимание полного цикла бэкенд-разработки — от анализа требований и проектирования архитектуры до реализации, тестирования и развёртывания.
